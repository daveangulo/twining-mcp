[example]
**Good:**

## Quality Criteria

### quality: kebab-case-name
| Level | Description | Example |
|-------|-------------|---------|
| good | ... | ... |
```

**Content requirements:**

1. **All 32 tools must appear** in the Tool Behaviors section, organized by category (Blackboard, Decisions, Context, Graph, Coordination, Lifecycle, Verification, Export).

2. **Tiering (per RESEARCH):**
   - Tier 1 (~10 tools): twining_assemble, twining_post, twining_decide, twining_why, twining_verify, twining_handoff, twining_status, twining_add_entity, twining_add_relation, twining_read
   - Each Tier 1 tool gets: Context, Rules table, Correct Usage JSON, Incorrect Usage JSON with explanation
   - Tier 2 (~22 tools): all remaining tools
   - Each Tier 2 tool gets: Context (1-2 sentences), Rules table (at minimum 1 rule), no usage examples required

3. **MUST rule hard cap: 8-12 total across all tools** (STATE.md decision). Draft all candidate MUST rules first, then select the top 8-12 by severity (prioritize rules preventing data corruption, coordination failures, or silent information loss). Demote remaining to SHOULD. Suggested allocation:
   - twining_decide: 2-3 MUSTs (rationale required, alternatives required, use twining_decide not twining_post for decisions)
   - twining_post: 1-2 MUSTs (entry_type must match content, scope must be specific)
   - twining_assemble: 1 MUST (call before making decisions in new scope)
   - twining_handoff: 1-2 MUSTs (include context snapshot, target must be registered)
   - twining_verify: 1 MUST (run before declaring work complete)
   - Remaining tools: SHOULD rules only

4. **Parameter names MUST match actual Zod schemas** from src/tools/*.ts. Cross-reference every usage example against the real tool registration. Do NOT invent parameter names.

5. **8+ workflow scenarios** derived from the 8 existing plugin skills:
   - orient, decide, verify, handoff, coordinate, map, review, dispatch
   - Add 2 composite scenarios if natural (e.g., "new-session-lifecycle", "conflict-resolution")
   - Each workflow: table with Step | Tool | Purpose columns

6. **Anti-pattern catalog** (>= 4 entries):
   - fire-and-forget-decisions (deciding without linking commits)
   - scope-inflation (using "project" when more specific exists)
   - rationale-poverty (vague rationale lacking specificity)
   - blackboard-spam (posting low-value entries that clutter context)
   - Each: Description, Bad example, Good example

7. **Quality criteria** (>= 3 entries):
   - scope-precision: good/acceptable/bad levels for scope specificity
   - rationale-quality: good/acceptable/bad levels for rationale content
   - parameter-content: good/acceptable/bad levels for tool parameter quality
   - Each: table with Level | Description | Example columns

**Derivation sources (in priority order):**
1. Actual tool registrations in src/tools/*.ts (parameter names, descriptions, Zod constraints)
2. Existing plugin SKILL.md files (workflow patterns, anti-patterns, when-to-use guidance)
3. TWINING-DESIGN-SPEC.md (architectural intent)
4. Research recommendations from 15-RESEARCH.md
