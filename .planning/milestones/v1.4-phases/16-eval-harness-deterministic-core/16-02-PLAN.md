  <behavior>
    Sequencing scorer:
    - Passes when tool calls follow workflow step order from BehaviorSpec workflows
    - Fails when calls are out of order (e.g., twining_decide before twining_assemble)
    - Checks against all matching workflows in the spec
    - Returns check per matched workflow with rule ID referencing the workflow name

    Scope quality scorer:
    - Fails (SHOULD level) when any tool call has scope argument of "project" or overly broad path
    - Passes when scopes look like file paths (contain "/" or ".")
    - Checks scope field in arguments of relevant calls (twining_post, twining_assemble, etc.)

    Argument quality scorer:
    - Fails (MUST level) when twining_decide call has no rationale argument or empty rationale
    - Fails (SHOULD level) when summary fields exceed 200 chars
    - Passes when arguments are populated with specific content
    - Checks tool-specific field requirements from BehaviorSpec rules

    Decision hygiene scorer:
    - Fails (MUST level) when twining_decide present without prior twining_assemble (blind-decisions)
    - Fails (SHOULD level) when twining_decide not followed by twining_link_commit (fire-and-forget)
    - Fails (MUST level) when twining_decide has no alternatives_considered
    - Passes when decide calls have assemble before and link_commit after

    Workflow completeness scorer:
    - Checks that recognized workflow patterns include all expected steps
    - If orient pattern detected (status call), checks for assemble and optionally why
    - If decide pattern detected, checks for preceding assemble
    - Returns checks per workflow found in the call sequence

    Anti-patterns scorer:
    - Detects fire-and-forget (decide without link_commit)
    - Detects scope-inflation (broad scope values)
    - Detects rationale-poverty (short/vague rationale text)
    - Detects blind-decisions (decide without assemble)
    - Each detection maps to an anti-pattern ID from BehaviorSpec

    Quality criteria scorer:
    - Checks scope-precision: scope values are narrow/specific
    - Checks rationale-quality: rationale text is substantive (>30 chars, contains specifics)
    - Checks parameter-content: required fields populated meaningfully
    - Each criterion maps to quality criteria from BehaviorSpec
  </behavior>
  <action>
    Create the test/eval/scorers/ directory with 7 scorer files, an index, and a test file.

    **Each scorer file** follows this pattern:
    - Import Scorer, ScorerResult, CheckResult, aggregateChecks, DEFAULT_THRESHOLD from "../scorer-types"
    - Import ScorerInput from "../scenario-schema"
    - Import BehaviorSpec (and sub-types as needed) from "../types"
    - Export a const implementing the Scorer interface
    - The score() method: builds CheckResult[] by running inline check functions, calls aggregateChecks to get numeric score, returns ScorerResult with passed = score >= DEFAULT_THRESHOLD
    - Check functions reference BehaviorSpec for rule metadata (IDs, levels) where applicable
    - Check functions implement the actual check logic inline (not delegates to BehaviorSpec)

    **Specific scorer implementations:**

    1. **sequencing.ts** - For each workflow in spec.workflows, check if the scenario's tool calls that match workflow step tools appear in the correct relative order. Not all workflow steps need to be present, but those that ARE present must be in order.

    2. **scope-quality.ts** - Find all tool calls with a "scope" argument. Check each: fail if scope is "project", ".", or "/" (too broad). Pass if scope contains a meaningful path segment. Map to rules from tools that have scope-related rules (POST, ASSEMBLE, etc.).

    3. **argument-quality.ts** - For twining_decide calls: check rationale present and non-empty (MUST per DECIDE-01), check alternatives_considered present (MUST per DECIDE-02). For twining_post calls: check summary under 200 chars (SHOULD). For twining_entity calls: check name is specific (MUST per ENTITY-01). Map rule IDs from spec.tools rules.

    4. **decision-hygiene.ts** - For each twining_decide call: check if twining_assemble appears earlier in sequence (MUST per ASSEMBLE-01, anti-pattern: blind-decisions). Check if twining_link_commit appears later (SHOULD, anti-pattern: fire-and-forget). Check alternatives_considered field populated (MUST per DECIDE-02).

    5. **workflow-completeness.ts** - Detect which workflow patterns are represented in the call sequence by checking if the first tool of any workflow appears. For each detected workflow, check how many of its expected steps appear. Score based on completeness ratio.

    6. **anti-patterns.ts** - Check for each anti-pattern in spec.antiPatterns by ID:
       - "fire-and-forget-decisions": twining_decide without twining_link_commit following
       - "scope-inflation": any scope arg matching overly-broad patterns
       - "rationale-poverty": twining_decide with rationale < 20 chars or generic text
       - "blind-decisions": twining_decide without preceding twining_assemble
       - "blackboard-spam": more than 5 twining_post calls in sequence (SHOULD)
       For each anti-pattern found: check fails. Anti-pattern absent: check passes.

    7. **quality-criteria.ts** - For each quality criterion in spec.qualityCriteria:
       - "scope-precision": check scope args for path specificity (SHOULD)
       - "rationale-quality": check rationale text length and specificity (SHOULD)
       - "parameter-content": check that tool-specific required fields are non-empty (SHOULD)
       - "alternative-depth": check alternatives_considered has 2+ items or substantive text (SHOULD)
