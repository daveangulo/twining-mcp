  <action>
Follow a strict iterative tuning process. Read TUNING-LOG.md (from Plan 19-01) to see the failure classification. Focus ONLY on "unexpected failures" (not expected negatives).

**Process for each workflow batch:**

1. **Read TUNING-LOG.md** to identify unexpected failures for this workflow category
2. **Read the relevant plugin skill file** to understand current instructions
3. **Read the failing scenario YAML** to understand what behavior is expected
4. **Read the scorer that's failing** to understand what it checks
5. **Make a targeted change** to the skill file that addresses the specific gap
6. **Run the full eval suite**: `npm run eval:synthetic`
7. **Check for regressions**: Compare results against pre-change state. If any previously passing scenario now fails, REVERT the change and try a different approach
8. **Run token budget check**: `bash scripts/measure-plugin-tokens.sh`
9. **Record in TUNING-LOG.md**: Add a tuning entry with date, workflow batch, target failure, change made (file + what + why), before/after scores, regression check result
10. **Run holdout validation**: `npm run eval:holdout` after completing each workflow batch

**Workflow order** (tackle by failure density, per Claude's discretion from CONTEXT.md):

Start with whichever workflow has the most unexpected failures based on the failure classification in TUNING-LOG.md. Typical order based on research analysis:

1. **Verify workflow** -- verify-happy-path/workflow-completeness(0.6667) is a clear happy-path failure
2. **Orient workflow** -- orient-minimal/wf-completeness(0.5), cross-full-lifecycle/sequencing(0.75)
3. **Coordinate workflow** -- coordinate-minimal/wf-completeness(0.5), coordinate-no-capability-check/sequencing(0.5)
4. **Handoff workflow** -- handoff-without-verify/sequencing+wf-completeness
5. **Decide workflow** -- most failures here are expected negatives, but decide-without-alternatives has unexpected argument-quality+decision-hygiene failures

**Types of plugin changes** (from specific to general):

a. **Add explicit workflow step instructions**: "ALWAYS call twining_assemble before twining_decide" style rules in SKILL.md
b. **Clarify parameter quality expectations**: Add examples of good vs bad arguments
c. **Add cross-skill workflow references**: "After completing orient, the next step is usually decide or verify"
d. **Move shared patterns to agent prompts**: If multiple skills repeat the same instruction, move to twining-aware-worker.md

**Critical constraints:**
- Token budget: Check `bash scripts/measure-plugin-tokens.sh` after each change. If approaching 41,806 bytes, restructure (move shared patterns to agent-level prompts)
- No scenario/scorer changes unless there's a genuine bug (document justification if you must)
- Prefer concise rule additions over verbose explanations
- Keep changes principle-based, not pattern-matched to specific scenarios

**When a failure is actually a threshold issue, not a plugin issue:**

Some failures (like verify-happy-path/workflow-completeness at 0.6667) may need the scenario or scorer adjusted rather than the plugin. If after reading the scenario and scorer you determine the scorer is too strict for a legitimate edge case:
1. Document the justification in TUNING-LOG.md
2. Fix the scenario (add missing steps) or adjust expected_scores
3. Mark as "scenario fix" not "plugin tuning"

**Target:** After all workflow batches, `npm run eval:synthetic` should show >= 95% pass rate (per locked decision).
